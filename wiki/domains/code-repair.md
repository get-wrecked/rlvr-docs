## Overview

Code repair (also called automated program repair or bug fixing) tasks the model with taking buggy source code — which fails one or more tests — and producing a corrected version that passes the full test suite. The verification mechanism is identical to code generation (run tests, check pass), but the input includes the broken code, error messages, and sometimes a failing test trace, making the task more constrained and arguably more realistic than generation from scratch.

This domain is a natural RLVR target because (a) the verifier is the same execute-and-test pipeline, (b) real-world bug datasets already exist with regression test suites, and (c) the problem is practically valuable — LLM-based repair has direct applications in developer tools and CI/CD.

## Verification Mechanism

```
def verify_repair(buggy_code, model_patch, test_suite, project_env):
    # 1. Apply the model's patch to the buggy code
    try:
        repaired_code = apply_patch(buggy_code, model_patch)
    except PatchApplicationError:
        return 0.0  # reward 0 if patch doesn't apply cleanly

    # 2. Build the project (for compiled languages)
    build_result = run_in_sandbox(
        command="build_project",
        cwd=project_env,
        timeout=120
    )
    if build_result.returncode != 0:
        return 0.0  # doesn't compile

    # 3. Run full test suite
    results = []
    for test in test_suite:
        try:
            outcome = run_in_sandbox(
                command=f"run_test {test.name}",
                cwd=project_env,
                timeout=30
            )
            results.append(outcome.returncode == 0)
        except TimeoutError:
            results.append(False)

    # 4. Reward: all tests pass (including previously-passing ones)
    #    This ensures the fix doesn't introduce regressions
    if all(results):
        return 1.0
    # Optional partial credit: fraction of previously-failing tests now fixed
    # without breaking previously-passing tests
    newly_fixed = sum(r and not orig for r, orig in zip(results, original_results))
    no_regressions = all(r for r, orig in zip(results, original_results) if orig)
    if no_regressions:
        return newly_fixed / num_originally_failing
    return 0.0
```

Key details:

- **Regression checking**: The repaired code must pass ALL tests — not just the ones that were originally failing. Breaking previously-passing tests yields reward 0.
- **Patch format**: Models can output unified diffs, search-and-replace blocks, or full file rewrites. Unified diff is preferred for multi-file patches.
- **Build verification**: For Java/C++ projects, the code must compile before tests can run. Compilation failure is an immediate 0.
- **Semantic equivalence**: Some setups additionally check that the fix is "minimal" (small diff from original), though this is harder to verify automatically and is usually a soft penalty rather than hard gate.

For RL training at scale, synthetic bugs can be generated by:
1. Taking working code and introducing mutations (variable renaming, off-by-one, wrong operator, deleted lines).
2. Mining version control history for bug-fix commits and extracting the pre-fix version.
3. Using models to introduce plausible bugs into known-correct code.

## Task Format

**Input prompt**:
```
The following code has a bug. The test `test_edge_case_empty_list`
is failing with "IndexError: list index out of range".

```python
def find_median(nums):
    nums.sort()
    mid = len(nums) // 2
    if len(nums) % 2 == 0:
        return (nums[mid] + nums[mid - 1]) / 2
    return nums[mid]
```

Fix the bug so all tests pass.
```

**Expected output** (diff or complete fixed code):
```python
def find_median(nums):
    if not nums:
        raise ValueError("Cannot find median of empty list")
    nums.sort()
    mid = len(nums) // 2
    if len(nums) % 2 == 0:
        return (nums[mid] + nums[mid - 1]) / 2
    return nums[mid]
```

**Hidden test suite** validates both the fix and non-regression.

## Difficulty Curriculum

| Level | Bug Type | Example | Typical Fix Size |
|-------|----------|---------|-----------------|
| 1 | Syntax errors | Missing colon, unmatched parenthesis | 1 character |
| 2 | Simple logic | Off-by-one, wrong comparison operator | 1–2 lines |
| 3 | Missing edge case | Null/empty input not handled | 2–5 lines |
| 4 | Algorithmic error | Wrong algorithm choice, incorrect DP transition | 5–20 lines |
| 5 | Multi-location | Bug requires changes in multiple functions | 10–50 lines across files |
| 6 | Architectural | Design flaw requiring structural changes | 50+ lines, multi-file |

SWE-bench sits at levels 4–6, while QuixBugs and FixEval mostly occupy levels 2–3.

## Limitations & Risks

- **Overfitting to test suites**: A model might produce patches that pass all tests but are semantically wrong (e.g., special-casing test inputs). Strong test suites mitigate this.
- **Environment complexity**: SWE-bench-style tasks require setting up full project environments with dependencies, which is expensive and fragile. Docker snapshots help.
- **Patch plausibility**: An RL-trained model might learn to produce "ugly" patches (massive rewrites) that pass tests. Adding a diff-size penalty to the reward can help but complicates the reward signal.
- **Non-deterministic tests**: Flaky tests in real-world projects cause noisy reward signals. Filtering or re-running tests mitigates this.
- **Evaluation expense**: Each episode requires building a project and running a test suite, which can take minutes per sample. Parallelization across many containers is essential.
