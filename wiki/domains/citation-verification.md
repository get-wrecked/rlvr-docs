# Citation Verification

## Overview

Citation verification tasks require the model to determine whether a claim is supported by a specific cited source. Given a claim and a source document (or passage), the model must judge whether the source actually supports the claim. This is crucial for reducing hallucination and improving attribution in LLM outputs.

This domain combines natural language inference with source retrieval. The verification mechanism checks the model's support/refute judgment against human-annotated gold labels, making it amenable to RLVR.

## Verification Mechanism

**Verification type: Exact**

The primary verification is label comparison:

```python
def verify(prediction: str, gold_label: str) -> int:
    # Labels: SUPPORTS, REFUTES, NOT_ENOUGH_INFO
    return 1 if normalize(prediction) == normalize(gold_label) else 0
```

For tasks that also require evidence selection (identifying which sentences in the source support/refute the claim), verification additionally checks:

```python
def verify_with_evidence(
    pred_label: str, pred_evidence: list[int],
    gold_label: str, gold_evidence: list[int]
) -> int:
    label_correct = normalize(pred_label) == normalize(gold_label)
    evidence_correct = set(pred_evidence) == set(gold_evidence)
    return 1 if label_correct and evidence_correct else 0
```

**Note on NLI-model-based verification**: Some citation verification setups use a trained NLI model as the verifier (e.g., checking if a generated claim is entailed by the cited passage). This blurs the line between exact and model-based verification. For strict RLVR, human-annotated gold labels are preferred over NLI model scores.

| HealthVer | 14K claims | Health/medical | Verify health claims | COVID-19 and general health |

**Scalability**: Moderate. Gold-annotated citation verification data is expensive to produce because annotators must read both the claim and the source. However, synthetic data can be generated by:
1. Extracting claims from papers and pairing with cited sources (positive examples)
2. Pairing claims with random unrelated sources (negative examples)
3. Mutating supported claims to create refuted examples

Scientific papers on Semantic Scholar, PubMed, and arXiv provide millions of citation-context pairs for silver data generation.

## Task Format

**Input**: A claim and a source passage (or document).

**Output**: A judgment label (SUPPORTS / DOES NOT SUPPORT / PARTIALLY SUPPORTS) and optionally the relevant evidence sentences.

**Prompt template example**:
```
Determine whether the following claim is supported by the cited source.

Claim: {claim}

Cited source: {source_passage}

Is the claim supported by the source? Answer: SUPPORTS, REFUTES, or NOT ENOUGH INFO.

Answer:
```

For evidence selection:
```
Given the claim and source below, identify which sentence(s) in the source are relevant to the claim, and determine whether the source supports or refutes the claim.

Claim: {claim}
Source:
[1] {sentence_1}
[2] {sentence_2}
...

Relevant sentences: [list sentence numbers]
Verdict: SUPPORTS / REFUTES / NOT ENOUGH INFO
```

## Difficulty Curriculum

1. **Level 1 — Direct support**: The source explicitly states the claim almost verbatim. Trivial entailment.
2. **Level 2 — Paraphrased support**: The source supports the claim but with different wording. Requires paraphrase recognition.
3. **Level 3 — Partial support**: The source supports part of the claim but not all of it. Requires fine-grained analysis.
4. **Level 4 — Numerical/statistical claims**: "The study found a 15% improvement..." Requires precise numerical comparison with source data.
5. **Level 5 — Negation and contradiction**: The source directly contradicts the claim. Requires detecting negation and semantic opposition.
6. **Level 6 — Multi-sentence evidence**: Support comes from combining information across multiple sentences in the source.
7. **Level 7 — Subtle misrepresentation**: The claim slightly distorts the source (cherry-picking, mischaracterizing scope, conflating correlation and causation). These are the hardest and most practically important cases.

## Limitations & Risks

- **Scale limitation**: High-quality gold data is limited (SciFact has only 1.4K claims). This constrains RLVR training.
- **Domain specificity**: Scientific citation verification requires domain expertise that may not transfer to news or legal citations.
- **Partial support ambiguity**: Many real-world citations provide partial support, which does not fit cleanly into SUPPORTS/REFUTES/NEI categories.
- **Source availability**: Verification requires access to the full cited source. Paywalled or unavailable sources cannot be verified.
- **NLI model dependency**: If using an NLI model as verifier (rather than gold labels), the reward inherits the NLI model's errors and biases. This is effectively LLM-as-judge, which weakens the RLVR paradigm.
- **Annotation quality**: Citation verification annotations require expert judgment (especially in scientific domains), and inter-annotator agreement can be moderate.
